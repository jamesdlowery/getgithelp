Install GitHub CLI (gh) local
What is gh? The official GitHub CLI. It talks to the GitHub platform — creating repos, opening PRs, managing issues — things git itself knows nothing about.
sudo apt install gh
Debian / Ubuntu / Mint
sudo dnf install gh

gh auth login handles its own auth separately from git's credential helper. You still need to set up git auth (PAT or SSH key) for git push / git pull operations.
2
Global Identity & Config run once
git config --global user.name "Your Name"
Your display name — appears in all commits
git config --global user.email "[email protected]"
Must match your GitHub account email
git config --global init.defaultBranch main
Set default branch name to main for all new repos
git config --global core.editor nano
Set default editor for commit messages
git config --global pull.rebase true
Default to rebase instead of merge on git pull
git config --global color.ui auto
Enable coloured terminal output
git config --global --list
Review all global settings
Global config is stored in ~/.gitconfig. Edit directly with nano ~/.gitconfig.
3
Create / Configure GitHub Account web UI
Sign up or log in at github.com
Use the same email as user.email above
Settings → Emails → verify your email address
GitHub won't accept pushes from an unverified email
Settings → Profile → add name, avatar (optional)
Helps collaborators identify you in PRs and reviews
Settings → Security → enable two-factor authentication (2FA)
Required for most organisations; strongly recommended for all
GitHub no longer accepts plain passwords for Git operations. You must authenticate using a Personal Access Token (HTTPS) or an SSH key.
4
Choose an Authentication Method
Personal Access Token (PAT) — HTTPS
Simpler to set up. Token acts as your password. Works anywhere. Best for getting started quickly.
SSH Key — SSH
More secure. No token to expire or rotate. Password-free after setup. Preferred for daily dev work.
You only need one method. Use PAT for simplicity; SSH for long-term use.
5
Option A — Personal Access Token HTTPS
Generate the token web UI
GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token
Set expiry; tick scopes: repo, workflow
Copy the token immediately — it is only shown once
Store it in a password manager (e.g. KeePassXC, Bitwarden)
Store the token so Git remembers it local
git config --global credential.helper store
Save to ~/.git-credentials (plaintext — trusted machines only)
git config --global credential.helper 'cache --timeout=3600'
Cache in memory for 1 hour (more secure)
git config --global credential.helper /usr/share/doc/git/contrib/credential/libsecret/git-credential-libsecret
Use GNOME Keyring / libsecret (most secure)
After setting a credential helper, the first git push prompts for username + token. Git stores it automatically after that.
6
Option B — SSH Key SSH
ssh-keygen -t ed25519 -C "[email protected]"
Generate a new Ed25519 key pair (recommended algorithm)
eval "$(ssh-agent -s)"
Start the SSH agent in the current shell
ssh-add ~/.ssh/id_ed25519
Add your private key to the agent
cat ~/.ssh/id_ed25519.pub
Print your public key — copy the entire output
GitHub → Settings → SSH and GPG keys → New SSH key
Paste the public key, give it a title, save
Add this to ~/.ssh/config for automatic key loading on login:
Host github.com
  AddKeysToAgent yes
  IdentityFile ~/.ssh/id_ed25519
7
Verify the Connection local
ssh -T [email protected]
Expect: Hi <username>! You've successfully authenticated
git config --global --list
Confirm all global settings are correct
git clone [email protected]:<user>/<repo>.git
SSH clone — confirms end-to-end auth works
git clone https://github.com/<user>/<repo>.git
HTTPS clone — confirms PAT auth works
Run ssh -vT [email protected] for verbose output if the connection test fails.
8
GH_TOKEN — environment variable authentication
An alternative to gh auth login for non-interactive environments. Set GH_TOKEN to a GitHub PAT and gh uses it automatically. If both are present, GH_TOKEN takes precedence.
gh auth loginGH_TOKEN
How setInteractive promptEnvironment variable
Persists across sessionsYes — stored in ~/.config/gh/hosts.ymlNo — lives in memory only
Works without a humanNoYes — fully scriptable
Best forDeveloper workstationsCI/CD, scripts, Docker, automation
Configures git credential helperYes (HTTPS option)No — git push/pull needs separate auth
Session-only (current terminal)
export GH_TOKEN=<your-PAT>
Set token for this shell session only
gh auth status
Verify gh is picking up the token
Persistent in shell profile (personal machine)
echo 'export GH_TOKEN=<your-PAT>' >> ~/.bashrc
Append to shell profile — loads on every new session
source ~/.bashrc
Apply immediately in the current session
Never hard-code a real token in a script that gets committed to a repo. Use your CI/CD platform's secret injection instead.
In a GitHub Actions workflow
- name: Run gh command
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }} # auto-provided by Actions
  run: gh pr list
GH_TOKEN authenticates gh only. For git push / git pull over HTTPS, configure the credential helper separately, or use SSH keys.

0
git vs gh — which to use?
git only
Works with GitHub, GitLab, Bitbucket, and any Git host. Must create the remote repo manually (web UI or API) before pushing. Best for host-agnostic scripts or multi-platform teams.
gh + git
GitHub only. Creates the remote repo on GitHub directly from the terminal — no web UI needed. Faster end-to-end setup. Best for GitHub-only workflows.
1
Option A — Create with gh (GitHub only) gh
gh repo create <project-name> --public --clone
Create remote repo on GitHub and clone it locally in one command
cd <project-name>
Navigate into the cloned folder
gh repo create <name> --public --source=. --remote=origin --push
Create from an existing local folder, set remote, and push immediately
FlagDescription
--public / --privateSet repo visibility
--cloneClone the new repo locally after creating it
--source=.Use the current directory as the local source
--remote=originSet the new GitHub repo as the origin remote
--pushPush local commits to the remote immediately
--description "text"Add a repo description
--add-readmeInitialise with a README on GitHub
--gitignore <template>Add a .gitignore from a GitHub template (e.g. Node, Python)
gh repo create --clone gives you a fully connected local + remote repo in a single command — no git remote add needed.
2
Option B — Create with git only local
Create & initialize
mkdir <project-name>
Create a new project folder
cd <project-name>
Navigate into that folder
git init
Initialize an empty Git repo (creates .git/)
You will need to create the remote repo on GitHub (web UI or API) before you can push.
Add content
touch README.md
Create an empty README file
echo "# My Project" >> README.md
Append content to a file from the terminal
git status
See which files are untracked / modified
git add <file>
Stage a specific file
git add .
Stage all new and modified files in current directory
git add -p
Interactively stage specific hunks (partial add)
Create a .gitignore before git add . to exclude files like node_modules/ or .env.
Initial commit
git commit -m "Initial commit"
Commit staged files with a message
git log --oneline
Verify the commit was recorded
git branch -M main
Rename default branch to main (if needed)
Connect remote & push
git remote add origin <url>
Link local repo to a remote (GitHub, GitLab, etc.)
git remote -v
Verify the remote URL is set correctly
git push -u origin main
Push commits and set upstream tracking branch
-u (--set-upstream) is only needed on the first push. After that, git push is enough.
3
Quick Reference
Using gh (fastest):
gh repo create <project-name> --public --clone
cd <project-name>
touch README.md
git add . && git commit -m "Initial commit"
git push
Using git only:
mkdir <project> && cd <project>
git init
# --- create remote on GitHub first, then: ---
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin <url>
git push -u origin main

Clone a Repository local
git clone <url>
Clone a repo into a new folder (folder named after repo)

Clone into a specific folder name
git clone --depth 1 <url>
Shallow clone — latest snapshot only, no full history

Clone and immediately check out a specific branch
Get the URL from the GitHub/GitLab repo page — use HTTPS for simplicity, SSH for key-based auth.
2
Explore After Cloning local
cd <project>
Navigate into the cloned folder
git log --oneline -10
View the last 10 commits
git branch -a
List all local and remote branches
git remote -v
Confirm the remote URL (should be origin)
git status
Confirm you are on the default branch with a clean tree
3
Check Out a Branch local
git checkout <branch>
Switch to an existing local branch
git checkout -b <branch>
Create and switch to a new branch
git checkout -b <branch> origin/<branch>
Create a local branch tracking an existing remote branch
git switch <branch>
Modern alternative to git checkout <branch>
git switch -c <branch>
Modern alternative to git checkout -b <branch>
Always create a new branch before making changes — never work directly on main or dev.
4
Configure the Clone (If Needed) local
git config user.name "Your Name"
Set name for this repo only
git config user.email "[email protected]"
Set email for this repo only
git remote set-url origin <new-url>
Change the remote URL (e.g. HTTPS → SSH)
git remote add upstream <url>
Add a second remote (e.g. original repo when working from a fork)
Use --global to apply settings to all repos, or omit it to apply to this repo only.
5
Stay Up to Date local
git fetch origin
Download remote changes without touching your working tree
git pull
Fetch and merge the tracked remote branch into current branch
git pull --rebase
Fetch and rebase local commits on top of remote (cleaner history)
git pull origin <branch>
Pull a specific remote branch into the current branch
Run git fetch origin regularly to stay aware of remote changes without committing to a merge.
6
Quick Reference

git branch -a # see all available branches
git checkout -b <your-branch> # create a working branch
# --- make your changes, then sync ---
git fetch origin # stay up to date
git pull --rebase # incorporate remote changes

Check What Has Changed local

Overview of all tracked/untracked changes in working tree
git status -s
Short format — two-letter status codes per file
git diff
Show unstaged line-level changes

Status codes: ?? untracked  |  M modified  |  A staged new  |  D deleted  |  R renamed
2
Stage Changes local newchangeddeletedmoved
git add .
newchanged Stage all new and modified files recursively
git add <file>
newchanged Stage a specific file
git add <directory>/
newchanged Stage all changes inside a directory
git rm <file>
deleted Stage a deletion and remove file from disk
git rm --cached <file>
deleted Untrack a file but keep it on disk
git mv <old> <new>
moved Rename or move a file and stage it atomically
git add -A
newchangeddeleted Stage everything
git add -p
Interactively choose which hunks to stage
git add . stages new + modified but NOT deletions made with plain rm. Use git add -A to catch those too.
3
Commit local
git commit -m "message"
Commit staged changes with an inline message
git commit -am "message"
Stage all tracked changes and commit in one step
git commit --amend
Modify the most recent commit (message or staged content)
--amend rewrites history — only use it before pushing, never on commits already on the remote.
4
Pull Before You Push local
git fetch origin
Download remote changes without merging
git pull
Fetch and merge remote branch into current branch
git pull --rebase
Fetch and rebase local commits on top of remote (cleaner history)
Always pull before pushing to avoid rejected pushes due to diverged history.
5
Push to Remote local
git push
Push committed changes to the tracked remote branch
git push origin <branch>
Push to a specific branch on the remote
git push --tags
Push all local tags to the remote
6
Undo & Recover local
git restore <file>
Discard unstaged changes to a file (restores from last commit)
git restore --staged <file>
Unstage a file (keeps changes in working tree)
git stash
Temporarily shelve all uncommitted changes
git stash pop
Re-apply the most recently stashed changes
git revert <commit>
Create a new commit that undoes a previous one (safe for shared branches)
git restore permanently discards local changes — there is no undo. Use git stash instead if you are unsure.
7
Quick Reference
git status
git add -A # stage new + changed + deleted
git commit -m "describe your changes"
git pull --rebase # sync remote changes first

1
Clean Up & Push Your Branch local
git status
Confirm no uncommitted changes remain
git push -u origin <your-branch>
Push branch to remote and set upstream tracking
The -u flag is only needed the first time. Subsequent pushes just need git push.
2
Open a Pull Request web UI
GitHub: Repo → Pull Requests → New pull request
base: dev  |  compare: <your-branch>
GitLab: Repo → Merge Requests → New merge request
source: <your-branch>  |  target: dev
Write a clear PR title and description — what changed, why, and any testing notes. Assign reviewers and link related issues.
3
Address Review Feedback local
git add -A && git commit -m "address review feedback"
Commit changes requested by reviewers
git push
Push updates — PR refreshes automatically
Repeat commit + push as many times as needed. The PR always reflects the latest state of your branch.
4
Resolve Conflicts (If Flagged in the PR) local
git checkout dev && git pull --rebase origin dev
Sync dev locally first
git checkout <your-branch>
Switch back to your branch
git merge dev
Bring dev changes into your branch to surface conflicts
nano <conflicting-file>
Resolve <<< / === / >>> markers manually
git add -A && git commit && git push
Stage, complete merge commit, push — PR will show no conflicts
Every conflict marker must be removed before committing. Leaving one in will break the build.
<<<<<<< HEAD (dev)
  code from dev
=======
  code from your branch
>>>>>>> your-branch
Delete the markers, keep whichever code (or combination) is correct, then git add the file.
5
Merge the PR web UI
Click "Merge pull request" (GitHub) or "Merge" (GitLab)
Choose your merge strategy
Confirm the merge
PR is closed; branch is merged into dev on the remote
StrategyWhat it does
Merge commitKeeps full branch history with a merge commit
Squash and mergeCondenses all branch commits into one commit on dev
Rebase and mergeReplays commits linearly onto dev — no merge commit
Check your team's convention. Most teams standardise on one strategy.
6
Sync Local Dev After the Merge local
git checkout dev
Switch to local dev branch
git pull origin dev
Pull the merged state from remote into local dev
7
Clean Up local
git branch -d <your-branch>
Delete local branch (platform may auto-delete the remote)
git push origin --delete <your-branch>
Delete remote branch if not auto-deleted
git fetch --prune
Remove stale remote-tracking references locally
Use -D instead of -d to force-delete a branch not yet fully merged (use with care).
8
Quick Reference
# --- local: push your branch ---
git push -u origin <your-branch>
# --- web UI: open PR, assign reviewers ---
# --- web UI: iterate on feedback (push more commits as needed) ---
# --- local: if conflicts are flagged in the PR ---
git checkout dev && git pull --rebase origin dev
git checkout <your-branch>
git merge dev # resolve conflicts, then:
git add -A && git commit && git push
# --- web UI: merge the PR once approved ---
# --- local: sync and clean up ---
git checkout dev && git pull origin dev
git branch -d <your-branch>
git fetch --prune

dev
fully tested
→

review & approve
web UI

1
Pre-Flight Checks local
git checkout dev && git pull origin dev
Ensure local dev is fully up to date
git log origin/main..dev --oneline
List all commits on dev not yet in main — review before proceeding
git diff origin/main..dev --stat
Summary of files changed between dev and main
git status
Confirm dev is clean with no uncommitted changes
Never promote a dev branch that has failing tests, uncommitted changes, or unreviewed commits.
2
Open a Release PR: dev → main web UI
GitHub: Pull Requests → New pull request
base: main  |  compare: dev
GitLab: Merge Requests → New merge request
source: dev  |  target: main
Title the PR clearly, e.g. Release v1.2.0. Include a changelog. Assign senior reviewers — this is going to production.
3
Review, Approve & Merge the PR web UI
Reviewers approve the PR
At least one (preferably two) senior approvals required
All CI checks pass
Tests, linting, security scans — nothing merges red
Click "Merge pull request" → "Create a merge commit"
Use a merge commit to preserve the full dev history on main
Do not use squash or rebase merge strategies for a dev → main promotion. A merge commit clearly marks the release point in the history.
4
Tag the Release local
git checkout main && git pull origin main
Switch to main and sync the merged state
git tag -a v1.2.0 -m "Release v1.2.0"
Create an annotated tag at the current HEAD of main
git push origin v1.2.0
Push the tag to the remote
git tag -l
List all tags to verify
Use semantic versioning: vMAJOR.MINOR.PATCH. Tags trigger CI/CD pipelines and create release artifacts automatically.
IncrementWhen to use
MAJOR (v2.0.0)Breaking changes — not backwards compatible
MINOR (v1.2.0)New features — backwards compatible
PATCH (v1.2.1)Bug fixes — backwards compatible
5
Sync Dev With Main Post-Merge local
git checkout dev
Switch back to dev
git merge main
Bring the merge commit back into dev to keep histories aligned
git push origin dev
Push the updated dev to remote
Skipping this step causes dev to diverge from main over time, leading to messy future merges.
6
Emergency Rollback DANGER
git revert HEAD && git push origin main
Safe — creates a new revert commit, preserves full history
git checkout main && git reset --hard <prev-tag>
Destructive — hard reset to a previous tag
git push --force-with-lease origin main
Force-push after hard reset — only if branch protections allow it
Force-pushing to main is destructive and rewrites shared history. Always prefer git revert. Only use reset + force-push as a last resort with full team awareness.
