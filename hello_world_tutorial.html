Branch HierarchyAll work starts in dev. Changes promote to test via PR, build and acceptance-test the executables, then promote from test to prod via a second PR. Each branch has its own build trigger tag prefix: dev-v*, test-v*, prod-v*.
§ 2

GitHub Actions WorkflowsThree automated build pipelines — one per branch stage — triggered by branch-scoped tags

How It WorksPushing a branch-scoped tag (dev-v0.1, test-v0.1, prod-v0.1) triggers the corresponding GitHub Actions workflow. Each workflow builds three platform executables using PyInstaller on three runners (ubuntu, windows, macos) in parallel, then uploads them as artifacts (dev) or attaches them to a draft/published GitHub Release (test/prod).

Create the Workflows Directory

terminal / PowerShellAll Platforms
mkdir -p .github/workflows

build-dev.yml — Dev Build (Actions Artifacts)

.github/workflows/build-dev.ymlYAML
name: Build Dev Executables

on:
  push:
    tags:
      - 'dev-v*'          # triggers ONLY on dev-vX.X tags

jobs:
  build:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        include:
          - os: ubuntu-latest
            platform: linux
            ext: ''
          - os: windows-latest
            platform: windows
            ext: '.exe'
          - os: macos-latest
            platform: macos
            ext: ''

    runs-on: ${{ matrix.os }}

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install PyInstaller
        run: pip install pyinstaller

      - name: Extract version from tag
        shell: bash
        run: echo "VERSION=${GITHUB_REF_NAME#dev-}" >> $GITHUB_ENV

      - name: Build executable
        shell: bash
        run: |
          OUTNAME="hello_world_dev_${{ env.VERSION }}_${{ matrix.platform }}${{ matrix.ext }}"
          pyinstaller --onefile --name "$OUTNAME" hello_world_dev_${{ env.VERSION }}.py

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: hello_world_dev_${{ env.VERSION }}_${{ matrix.platform }}
          path: dist/hello_world_dev_*

§ 3

Issues & Project PlanningOpen an issue for every planned release before touching any code

Why Issues First?GitHub Issues create a trackable record of planned work. Linking commits and PRs to issues gives you a complete audit trail — from "we decided to add this greeting" to "it shipped to prod." Issues also auto-close when the prod PR merges, keeping your issue tracker clean automatically.

Opening a Release Issue

Use the Release issue template created in §1. Via the GitHub web UI or gh CLI:

gh CLIAll Platforms
gh issue create \
  --title "Release v1.1 — Hello, United States!" \
  --label "release" \
  --body "## Release Plan

**Version:** v1.1
**Greeting:** \`print(\"Hello, United States!\")\`
**Scope:** Country

## Checklist
- [ ] Dev: add greeting, test locally, tag dev-v1.1, test executables, PR dev→test
- [ ] Test: tag test-v1.1, acceptance test executables, PR test→prod
- [ ] Prod: tag prod-v1.1, publish release, sync branches, close issue"
Note the Issue NumberGitHub assigns a number to each issue (e.g., #7). You'll reference this in your commit messages and PRs to create automatic cross-links. Check the output of gh issue create or the GitHub web UI to find it.

Linking Issues to Commits and PRs

WhereSyntaxEffect
Commit messagerefs #7 or see #7Creates a cross-link — issue stays open
PR bodycloses #7Auto-closes the issue when the PR merges to the default branch (prod)
PR bodyfixes #7Same as closes
Importantcloses #N in a PR body only auto-closes the issue when the PR merges into the default branch (prod). Put it in the test→prod PR body, not the dev→test PR body, so it fires at the right moment.

§ 4

Commit Message ConventionsConventional Commits — structured, readable, automation-friendly

Why Conventional Commits?A consistent commit message format makes your history scannable at a glance, enables automated changelog generation, and signals intent clearly to reviewers and future maintainers. It also works with tools like semantic-release for automated versioning.

Format

structureAll commits
<type>(<scope>): <short description>

[optional body — wrap at 72 chars]

[optional footer — refs #N, closes #N, BREAKING CHANGE: ...]

Types Used in This Project

TypeWhen to UseExample
featA new greeting / featurefeat: add Hello United States (v1.1)
fixCorrecting a bug or wrong outputfix: correct spelling of Tuscaloosa
choreNon-code changes — workflows, templates, configchore: add GitHub Actions workflows
docsDocumentation only changesdocs: update README with v1.1 release info
ciChanges to CI/CD config filesci: pin PyInstaller to 6.x in build workflows
refactorCode change that neither fixes a bug nor adds a featurerefactor: extract greetings into a list
testAdding or correcting teststest: add output verification script

Examples from This Tutorial

good commit messagesExamples
# Adding a greeting (the most common commit in this project)
feat: add Hello God — initial hello_world_dev_v0.1.py

# Referencing the issue without closing it (dev→test PR merge)
feat: add Hello United States (v1.1) refs #7

# Closing the issue (goes in the test→prod PR body, not commit)
closes #7

# Chore commits
chore: add PR and issue templates
chore: add build-dev, build-test, build-prod GitHub Actions workflows
chore: sync dev and test branches to prod after v1.1 release
Keep the Subject Line Under 72 CharactersGit truncates longer lines in many views. If you need more detail, use the body — leave a blank line after the subject, then write as much as you need.

Closing PRs vs. Abandoning Them

If a PR needs to be abandonedClose the PR without merging (gh pr close <number>). Add a comment explaining why. Do NOT delete the branch from the remote — keep it for reference. Reopen a new PR if the work is resumed later.

Reviewing a PR

gh CLI — review commandsAll Platforms
# View open PRs
gh pr list

# Review a specific PR
gh pr view <number>

# Approve a PR
gh pr review <number> --approve --body "All checks pass. Looks good."

# Request changes
gh pr review <number> --request-changes --body "Please fix X before merging."

# Leave a comment without approving or blocking
gh pr review <number> --comment --body "Minor nit: ..."

Release Notes ReferenceWhat makes good release notes — and how to write them for this project

Why Release Notes MatterRelease notes are the public record of what changed, why it changed, and how to use the new version. For this tutorial project they're intentionally lightweight — but the structure is the same as any real-world project.

Structure for This Project

release notes templateAll Releases
## vX.X — Hello, [Scope]!

### What's New
- Added `print("Hello, [Scope]!")` — [one sentence about the scope]

### Usage
```
./hello_world_prod_vX.X_linux          # Linux
 .\hello_world_prod_vX.X_windows.exe   # Windows
./hello_world_prod_vX.X_macos          # macOS
```

### Downloads
Three platform executables are attached to this release:
- hello_world_prod_vX.X_linux
- hello_world_prod_vX.X_windows.exe
- hello_world_prod_vX.X_macos

### Output (cumulative — all lines printed in order)
Hello, God!
...
Hello, [Scope]!          ← new in this release

No external dependencies. Requires no Python installation.

Do's and Don'ts

DoDon't
State what changed, plainlyUse jargon or internal shorthand
Show the new expected outputAssume the reader knows the previous state
List all downloadable assets explicitlyMake users guess which file to download
Link to the closing issueLeave the connection to the issue tracker invisible
Note any breaking changes prominentlyBury breaking changes in the middle of a paragraph
